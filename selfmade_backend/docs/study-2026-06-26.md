### 2. DTO 분리

**개념**: 단일 `UserDto`를 역할에 맞게 3개로 분리

**왜 필요한지**: Request와 Response의 역할이 다르고, Entity를 직접 노출하면 보안 위험(비밀번호 등)이 있기 때문

| 클래스 | 역할 |
|--------|------|
| `SignUpRequestDto` | 회원가입 요청 (name, email, password + 검증 어노테이션) |
| `LoginRequestDto` | 로그인 요청 (email, password) |
| `UserResponseDto` | 응답 (id, name, email — password 제외) |
| `TokenResponseDto` | 토큰 응답 (accessToken, refreshToken) |

```java
// SignUpRequestDto - toEntity() 포함
public User toEntity(String encodedPassword) {
    return User.builder()
            .name(this.name)
            .email(this.email)
            .password(encodedPassword)
            .build();
}

// UserResponseDto - 정적 팩토리 메서드
public static UserResponseDto from(User user) {
    return UserResponseDto.builder()
            .id(user.getId())
            .name(user.getName())
            .email(user.getEmail())
            .build();
}
```

### 3. 전역 예외 처리

**개념**: `@RestControllerAdvice`로 예외 처리 로직을 한 곳에 모으는 것

**왜 필요한지**: 예외 처리 중앙화, 일관된 에러 응답 포맷, Controller 코드 간결화

**예외 계층 구조**:
```
BusinessException (최상위 커스텀 예외)
├── DuplicateEmailException    → 409 Conflict
├── UserNotFoundException      → 404 Not Found
├── InvalidPasswordException   → 401 Unauthorized
└── InvalidTokenException      → 401 Unauthorized
```

**각 계층의 책임**:
- **Service** — 비즈니스 규칙 위반을 감지하고 예외를 던지는 역할
- **GlobalExceptionHandler** — 던져진 예외를 잡아서 HTTP 응답으로 변환하는 역할
- **Controller** — 성공했을 때 응답을 만드는 역할

```java
@Slf4j
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(BusinessException.class)
    public ResponseEntity<ApiResponse<Void>> handleBusinessException(BusinessException e) {
        log.warn("Business exception: {}", e.getMessage());
        return ResponseEntity.status(e.getStatus())
                .body(ApiResponse.error(e.getStatus().value(), e.getMessage()));
    }
}
```

### 4. JwtProvider

**개념**: JWT 토큰 생성 / 검증 / 파싱을 담당하는 유틸 클래스

**왜 필요한지**: 토큰 관련 로직을 한 곳에 모아서 재사용하기 위해

| 메서드 | 역할 |
|--------|------|
| `generateAccessToken()` | 액세스 토큰 생성 (30분) |
| `generateRefreshToken()` | 리프레시 토큰 생성 (7일) |
| `getEmail()` | 토큰에서 이메일 추출 |
| `isValid()` | 토큰 유효성 검증 |

```java
private String generateToken(String email, long expiration) {
    Date now = new Date();
    return Jwts.builder()
            .subject(email)
            .issuedAt(now)
            .expiration(new Date(now.getTime() + expiration))
            .signWith(secretKey)
            .compact();
}
```

### 5. SecurityConfig

**개념**: Spring Security 전반적인 설정을 담당하는 클래스

```java
http
    .csrf(AbstractHttpConfigurer::disable)        // JWT는 CSRF 불필요
    .sessionManagement(session ->
        session.sessionCreationPolicy(SessionCreationPolicy.STATELESS)  // 세션 미사용
    )
    .authorizeHttpRequests(auth -> auth
        .requestMatchers("/users/signup", "/users/login", "/users/tokens/reissue").permitAll()
        .anyRequest().authenticated()
    )
    .addFilterBefore(new JwtAuthFilter(jwtProvider),
            UsernamePasswordAuthenticationFilter.class);
```

**주요 개념 정리**:
- **CSRF 비활성화**: JWT는 `Authorization` 헤더로 인증하므로 CSRF 공격 위험 없음
- **STATELESS**: 서버가 세션을 만들지 않음. JWT 방식은 토큰으로만 인증
- **addFilterBefore**: `UsernamePasswordAuthenticationFilter` 앞에 `JwtAuthFilter` 등록

### 6. JwtAuthFilter

**개념**: 모든 요청이 Controller에 도달하기 전에 JWT를 검증하는 필터

**왜 필요한지**: 토큰을 발급만 하고 검증하지 않으면 보호된 API가 누구나 접근 가능

```java
@Override
protected void doFilterInternal(HttpServletRequest request,
                                HttpServletResponse response,
                                FilterChain filterChain) throws ServletException, IOException {

    String token = resolveToken(request);

    if (token != null && jwtProvider.isValid(token)) {
        String email = jwtProvider.getEmail(token);
        UsernamePasswordAuthenticationToken authentication =
                new UsernamePasswordAuthenticationToken(email, null, List.of());
        SecurityContextHolder.getContext().setAuthentication(authentication);
    }

    filterChain.doFilter(request, response);  // 반드시 호출 (안 하면 요청이 멈춤)
}
```

**핵심 포인트**:
- `OncePerRequestFilter` — 하나의 요청에 필터가 한 번만 실행되도록 보장
- `UsernamePasswordAuthenticationToken` — Spring Security가 인식하는 인증 증명서
- `filterChain.doFilter()` — 다음 필터로 넘기는 것. 반드시 무조건 호출해야 함

### 7. 회원가입 / 로그인 구현

**회원가입 흐름**:
```
이메일 중복 체크 → 비밀번호 BCrypt 암호화 → DB 저장 → UserResponseDto 반환
```

**로그인 흐름**:
```
이메일로 유저 조회 → 비밀번호 검증 → 액세스 토큰 + 리프레시 토큰 발급 → Redis에 리프레시 토큰 저장
```

```java
@Transactional(readOnly = true)
public TokenResponseDto login(LoginRequestDto request) {
    User user = userRepository.findByEmail(request.getEmail())
            .orElseThrow(UserNotFoundException::new);

    if (!passwordEncoder.matches(request.getPassword(), user.getPassword())) {
        throw new InvalidPasswordException();
    }

    String accessToken = jwtProvider.generateAccessToken(user.getEmail());
    String refreshToken = jwtProvider.generateRefreshToken(user.getEmail());

    refreshTokenRepository.save(user.getEmail(), refreshToken,
            jwtProvider.getRefreshTokenExpiration());

    return TokenResponseDto.builder()
            .accessToken(accessToken)
            .refreshToken(refreshToken)
            .build();
}
```

### 8. Redis - Refresh Token 저장

**개념**: 리프레시 토큰을 Redis에 저장해서 TTL 기반 자동 만료 처리

**왜 Redis를 쓰나**:

| | DB 저장 | Redis 저장 |
|--|---------|-----------|
| 만료 처리 | 직접 삭제 필요 | TTL 자동 만료 |
| 조회 속도 | 느림 | 빠름 |
| 적합한 상황 | 소규모 | 실무, 대규모 |

**Key 구조**: `RT:siwoo@test.com` (prefix로 다른 데이터와 구분)

```java
public void save(String email, String refreshToken, long expiration) {
    redisTemplate.opsForValue()
            .set(KEY_PREFIX + email, refreshToken, expiration, TimeUnit.MILLISECONDS);
    //           키                  값              TTL         단위
}
```

### 9. 토큰 재발급 (Refresh Token Rotation)

**개념**: 리프레시 토큰으로 새 액세스 토큰 발급 시, 리프레시 토큰도 함께 교체

**왜 Rotation이 필요해**: 같은 리프레시 토큰을 계속 쓰면 탈취됐을 때 7일 내내 악용 가능. 재발급마다 교체해서 이전 토큰을 무효화

**재발급 흐름**:
```
리프레시 토큰 유효성 검증
→ 토큰에서 email 추출
→ Redis에서 저장된 토큰 조회
→ 요청 토큰 == Redis 토큰 일치 확인 (불일치 시 401)
→ 새 액세스 토큰 + 리프레시 토큰 발급
→ Redis 덮어쓰기
```

**흐름**:
```
액세스 토큰으로 인증 → SecurityContext에서 email 꺼냄 → Redis에서 리프레시 토큰 삭제
```

## 🐛 문제 해결

### 문제 1 - Redis 포트 바인딩 실패
- **원인**: `${REDIS_PORT}` 환경변수가 주입이 안 되어 문자열로 읽힘
- **해결**: 기본값 추가 `${REDIS_PORT:6379}`

### 문제 2 - Redis 연결 실패
- **원인**: `REDIS_HOST=redis`로 설정되어 Docker 내부 네트워크 호스트명으로 연결 시도
- **해결**: `REDIS_HOST=localhost`로 수정 (Spring 앱은 Docker 밖에서 실행)

### 문제 3 - Postman 403 오류
- **원인**: `Inherit auth from parent`가 제대로 동작하지 않아 Authorization 헤더 미전송
- **해결**: 각 요청 Authorization 탭에서 `Bearer Token` / `{{accessToken}}` 직접 설정
