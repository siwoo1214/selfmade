### 2. API 설계란?

**개념**: 프론트가 백엔드한테 어떻게 요청을 보낼지 미리 약속하는 것.
컨트롤러가 어떤 URL로, 어떤 HTTP 메서드로, 어떤 데이터를 받고, 어떤 데이터를 돌려줄지 미리 정의해두는 작업.

**왜 필요한가**:
- 프론트-백엔드 동시 개발이 가능해짐
- 프론트는 API 문서 보고 Mock 데이터로 먼저 화면 개발 가능
- 백엔드는 API 문서 보고 컨트롤러/서비스 개발 가능
- 포트폴리오 관점에서 "설계부터 제대로 했다"는 인상을 줌

### 3. Phase 0 — 프로젝트 세팅

#### 3-1. ApiResponse 수정

**개념**: 모든 API 응답을 일관된 포맷으로 반환하기 위한 공통 응답 클래스.

**왜 필요한가**: 클라이언트(프론트)가 응답을 예측 가능한 형태로 받을 수 있어야 처리하기 편하다.
HTTP 상태 코드는 `ResponseEntity`가 담당하고, 응답 본문은 `ApiResponse`가 담당하도록 역할을 분리한다.

```java
@Getter
@NoArgsConstructor
public class ApiResponse<T> {
    private boolean success;
    private String message;
    private T data;

    public static <T> ApiResponse<T> success(T data) {
        ApiResponse<T> response = new ApiResponse<>();
        response.success = true;
        response.message = "요청이 성공했습니다.";
        response.data = data;
        return response;
    }

    public static <T> ApiResponse<T> success(T data, String message) {
        ApiResponse<T> response = new ApiResponse<>();
        response.success = true;
        response.message = message;
        response.data = data;
        return response;
    }

    public static <T> ApiResponse<T> error(String message) {
        ApiResponse<T> response = new ApiResponse<>();
        response.success = false;
        response.message = message;
        response.data = null;
        return response;
    }
}
```

**변경 포인트**:
- 기존 `status` 필드(숫자) 제거 → HTTP 상태 코드는 `ResponseEntity`가 담당
- `success` boolean 필드 추가 → 프론트에서 성공/실패 판단하기 쉬움
- `success(data, message)` 오버로딩 추가 → 성공 메시지를 커스텀하고 싶을 때 사용

#### 3-2. CORS란? (CorsConfig)

**개념**: Cross-Origin Resource Sharing — 다른 출처 간 자원 공유.

**출처(Origin)**: `프로토콜 + 도메인 + 포트` 의 조합.
```
http://localhost:3000   →  React 프론트
http://localhost:8080   →  Spring 백엔드
```
포트만 달라도 **다른 출처**다.

**왜 막히는가**: 브라우저에는 **Same-Origin Policy(동일 출처 정책)** 라는 보안 정책이 있다.
다른 출처로 요청을 보내는 건 기본적으로 막힌다.

> **중요**: Postman으로 테스트할 때 CORS 에러가 안 나는 이유 — Postman은 브라우저가 아니기 때문이다.

**CorsConfig가 하는 일**: 백엔드가 응답 헤더에 "이 출처는 허용한다"는 표시를 붙여줌.
브라우저가 그걸 보고 통과시켜주는 흐름.

```
① React(3000)  →  백엔드(8080) API 요청
② 백엔드       →  응답 + "localhost:3000은 허용됨" 헤더 포함
③ 브라우저     →  헤더 확인 후 통과 ✅
```

```java
@Configuration
public class CorsConfig {

    @Bean
    public CorsConfigurationSource corsConfigurationSource() {
        CorsConfiguration config = new CorsConfiguration();

        // 허용할 출처 (React 개발 서버)
        config.setAllowedOrigins(List.of("http://localhost:3000"));

        // 허용할 HTTP 메서드
        config.setAllowedMethods(List.of("GET", "POST", "PUT", "PATCH", "DELETE", "OPTIONS"));

        // 모든 요청 헤더 허용 (Authorization 헤더 포함)
        config.setAllowedHeaders(List.of("*"));

        // JWT를 Authorization 헤더로 보낼 수 있도록 허용
        config.setAllowCredentials(true);

        // 프론트가 응답에서 Authorization 헤더를 읽을 수 있도록 노출
        config.setExposedHeaders(List.of("Authorization"));

        UrlBasedCorsConfigurationSource source = new UrlBasedCorsConfigurationSource();
        source.registerCorsConfiguration("/**", config);

        return source;
    }
}
```

**SecurityConfig에서도 설정하는 이유**:
Spring Security가 자체적으로 요청을 먼저 가로챔.
CorsConfig만 있으면 Spring Security 필터가 먼저 요청을 막아버릴 수 있어서,
SecurityConfig에서 `.cors(cors -> cors.configurationSource(corsConfigurationSource))` 로 연결해줌.

- `CorsConfig` → CORS 정책 정의 (어떤 출처를 허용할지)
- `SecurityConfig` → Spring Security가 그 정책을 사용하도록 연결

### 4. Phase 1 — 인증 / 회원 URL 정비

**변경 내용**: 기존 URL에 `/api` prefix 추가, 회원가입 응답 코드 수정.

```java
@RestController
@RequiredArgsConstructor
@RequestMapping("/api/auth")    // /users → /api/auth 로 변경
public class UserController {

    @PostMapping("/signup")
    public ResponseEntity<ApiResponse<UserResponseDto>> signUp(@Valid @RequestBody SignUpRequestDto request) {
        UserResponseDto data = userService.signUp(request);
        return ResponseEntity.status(201).body(ApiResponse.success(data, "회원가입이 완료되었습니다."));
        // 201 Created — 리소스 생성 성공 시 RESTful 표준 응답 코드
    }

    @PostMapping("/login")
    public ResponseEntity<ApiResponse<TokenResponseDto>> login(@Valid @RequestBody LoginRequestDto request) {
        TokenResponseDto data = userService.login(request);
        return ResponseEntity.ok(ApiResponse.success(data));
    }

    @PostMapping("/logout")
    public ResponseEntity<Void> logout(@AuthenticationPrincipal String email) {
        userService.logout(email);
        return ResponseEntity.noContent().build();   // 204 No Content
    }
}
```

### 5. Phase 2 — 판매자 승인

#### 5-1. User 엔티티 수정

```java
@Getter
@Entity
@Table(name = "users")
@NoArgsConstructor(access = AccessLevel.PROTECTED)
public class User {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String name;

    @Column(unique = true)      // DB 레벨에서도 중복 방지
    private String email;

    private String password;
    private String phone;       // 추가

    @Enumerated(EnumType.STRING)
    private Role role;          // 추가

    @Builder
    public User(String name, String email, String password, String phone, Role role) {
        this.name = name;
        this.email = email;
        this.password = password;
        this.phone = phone;
        this.role = role;
    }
}
```

**`@Enumerated(EnumType.STRING)` 을 쓰는 이유**:
- `EnumType.ORDINAL` (기본값) — enum 순서(0, 1, 2)를 DB에 저장 → enum 순서가 바뀌면 데이터 꼬임
- `EnumType.STRING` — enum 이름("CUSTOMER", "SELLER")을 DB에 저장 → 안전하고 가독성 좋음

#### 5-2. Company 엔티티

**개념**: SELLER 유저의 기업 정보를 저장하는 엔티티.
가입 신청 시 `PENDING`, ADMIN 승인 시 `APPROVED`, 거절 시 `REJECTED` 로 상태 관리.

```java
@Getter
@Entity
@Table(name = "companies")
@NoArgsConstructor(access = AccessLevel.PROTECTED)
public class Company {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @OneToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "user_id")
    private User user;

    private String name;

    @Column(unique = true)
    private String businessNumber;

    private String representative;
    private String address;

    @Enumerated(EnumType.STRING)
    private CompanyStatus status;

    private LocalDateTime createdAt;

    @Builder
    public Company(User user, String name, String businessNumber,
                   String representative, String address) {
        this.user = user;
        this.name = name;
        this.businessNumber = businessNumber;
        this.representative = representative;
        this.address = address;
        this.status = CompanyStatus.PENDING;    // 가입 신청 시 항상 PENDING
        this.createdAt = LocalDateTime.now();
    }

    // 도메인 메서드 — Dirty Checking 활용
    public void approve() {
        this.status = CompanyStatus.APPROVED;
    }

    public void reject() {
        this.status = CompanyStatus.REJECTED;
    }
}
```

**`approve()`, `reject()` 도메인 메서드를 만드는 이유**:
- setter로 `setStatus(CompanyStatus.APPROVED)` 하는 것보다 **의도가 명확**함
- 상태 변경 로직이 엔티티 안에 캡슐화됨
- Dirty Checking이 동작해서 별도로 `save()` 호출 없이 자동 UPDATE

**Dirty Checking이란**:
JPA가 트랜잭션 내에서 엔티티의 변경을 감지해서 트랜잭션 커밋 시점에 자동으로 UPDATE 쿼리를 실행하는 기능.
영속 상태의 엔티티에서만 동작한다.

```java
// Dirty Checking 활용 예시
@Transactional
public void approve(Long companyId) {
    Company company = companyRepository.findById(companyId)
            .orElseThrow(CompanyNotFoundException::new);
    company.approve();   // 여기서 상태만 바꿔줌
    // save() 호출 없어도 트랜잭션 커밋 시 자동 UPDATE ✅
}
```

#### 5-3. @RequestParam vs @RequestBody

**데이터가 어디에 담겨 오느냐**가 다르다.

| | `@RequestParam` | `@RequestBody` |
|--|----------------|----------------|
| 데이터 위치 | URL 뒤 (`?key=value`) | HTTP Body (JSON) |
| 주요 용도 | 필터, 검색, 페이징 | 생성, 수정 데이터 |
| 주로 쓰는 메서드 | `GET` | `POST`, `PUT`, `PATCH` |
| 데이터 노출 | URL에 노출됨 | 노출 안 됨 |
| 데이터 양 | 적은 데이터 | 많은 데이터 가능 |

```java
// @RequestParam — URL에 조건 담아서 조회
// GET /api/admin/companies?status=PENDING
@GetMapping("/api/admin/companies")
public ResponseEntity<?> getCompanies(
        @RequestParam(defaultValue = "PENDING") CompanyStatus status
) { ... }

// @RequestBody — Body에 데이터 담아서 생성
// POST /api/auth/signup + { "name": "홍길동", ... }
@PostMapping("/api/auth/signup")
public ResponseEntity<?> signUp(
        @RequestBody SignUpRequestDto request
) { ... }
```

#### 5-4. 판매자 가입 흐름

```
POST /api/auth/signup/seller
        ↓
User 저장 (role = SELLER)
        ↓
Company 저장 (status = PENDING)
        ↓
ADMIN이 GET /api/admin/companies?status=PENDING 으로 목록 조회
        ↓
PATCH /api/admin/companies/{id}/approve  →  Company.status = APPROVED
또는
PATCH /api/admin/companies/{id}/reject   →  Company.status = REJECTED
```

#### 5-5. JpaRepository를 상속받는 이유

```java
public interface CompanyRepository extends JpaRepository<Company, Long> { ... }
```

`JpaRepository<엔티티타입, PK타입>` 을 상속받으면 기본 CRUD 메서드가 자동으로 제공된다.

- `findById(Long id)` — 단건 조회
- `findAll()` — 전체 조회
- `save(entity)` — 저장 / 수정
- `delete(entity)` — 삭제
- `existsById(Long id)` — 존재 여부 확인

상속받지 않으면 이 메서드들을 직접 구현해야 한다.

### 6. Git 모노레포 구성

**문제**: `.git` 폴더가 `selfmade_backend` 안에 있어서 루트에서 `git add .` 하면 에러 발생.

**해결**: 루트에서 통합 관리하는 모노레포 구조로 변경.

```
selfmade/               ← 여기서 git 관리 (.git 위치)
├── selfmade_backend/
└── selfmade_front/
```

```bash
# 기존 .git 제거
cd selfmade_backend
Remove-Item -Recurse -Force .git

# 루트에서 git 초기화
cd ..
git init
git remote add origin https://github.com/siwoo1214/selfmade.git
git add .
git commit -m "chore: 프로젝트 초기 세팅 (백엔드 + 프론트)"
git branch -M main
git push -u origin main
```

**IntelliJ Git 루트 설정**:
`File → Settings → Version Control → Directory Mappings`
→ `selfmade/` 루트 경로로 변경해야 파일 변경 시 색깔 표시가 돌아옴.

## 🐛 문제 해결

### 문제 1 — ADMIN 계정 비밀번호 불일치
**상황**: DB에 BCrypt 해시값을 직접 넣었는데 로그인 시 비밀번호 불일치 에러.
**원인**: BCrypt 해시값은 환경마다 다르게 인식될 수 있음.
**해결**: 회원가입 API로 계정 생성 후 DB에서 role만 ADMIN으로 변경.
```sql
UPDATE users SET role = 'ADMIN' WHERE email = 'admin@test.com';
```

### 문제 2 — CompanyRepository에서 findById 빨간줄
**상황**: `CompanyService`에서 `companyRepository.findById()` 호출 시 빨간줄.
**원인**: `CompanyRepository`가 `JpaRepository`를 상속받지 않아서 기본 CRUD 메서드가 없었음.
**해결**: `extends JpaRepository<Company, Long>` 추가.

### 문제 3 — git push 에러 (src refspec main does not match any)
**상황**: `git push origin main` 실행 시 에러.
**원인**: `git add .`와 `git commit`을 하지 않은 상태에서 push 시도.
**해결**: main 브랜치로 이동 후 add → commit → push 순서대로 실행.
